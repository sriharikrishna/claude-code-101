Exercise 1 · framing

Why an FEM author needs CLAUDE.md

Numerical PDE work is full of recurring micro-decisions — small, stable choices that shape every result and that you don't want to re-justify in every prompt. CLAUDE.md is where those live.

DecisionWhy it belongs in CLAUDE.md
Mesh source and formatPinned upstream by your geometry pipeline. Letting the assistant pick a different format mid-project breaks reproducibility.
Element family and polynomial orderOne number with outsized effect on convergence rates. Easy to set wrong, easy to change silently.
Error norms reported (L², H¹, energy, …)The norm you report defines what "convergence" means for your paper. A study that swaps norms between runs is not a study.
Figure conventions for convergence plotsLog-log axes, reference slope lines, file format, output directory. Three of these wrong and the figure is unusable in a paper.
Sanity-check valuesEvery PDE solver should report a small number (max of solution, energy, integral) you can eyeball against a known reference. Pin it once and forget.

CLAUDE.md is read at the start of every session. The assistant doesn't re-derive your conventions — it inherits them.

Exercise 1 10 min

Write your own CLAUDE.md

  1. Open the exercise folder: cd exercises-pde/01-claude-md/
  2. Read laplace_square.py — a deliberately under-specified Firedrake solve on a coarse UnitSquareMesh(8, 8) with hardcoded P1 elements and no convergence study.
  3. Run claude on the host. First, with no CLAUDE.md, ask: add a convergence study for laplace_square.py. Note what Claude picks for norms, axes, file paths, and element-order handling.
  4. Now write a CLAUDE.md (use /init or copy the seed in README.md). Encode your conventions for: error norms (L² and H¹), table format, figure path and axes, reference slope lines, and how element order is exposed.
  5. /clear the conversation, re-ask the same prompt, compare.
Discussion: which conventions did Claude follow on the second try? Which did it miss? Add the missing rule to CLAUDE.md and ask once more — that's the whole loop.

Exercise 1 · debrief

What to look for in Claude's output

The improvement that respects a well-written CLAUDE.md should look something like this. Use it as a checklist, not an answer key.

Convention you wroteWhat it produces in the code
L² and H¹ errors with ratesA printed table with columns n, dofs, L2, L2 rate, H1, H1 rate. Rates computed as log2(prev/curr).
Log-log convergence plot with reference slopesTwo data lines (L², H¹) and two dotted reference lines at O(h^(p+1)) and O(h^p), anchored at the coarsest data point.
Figures saved, never shownNo plt.show(); output goes to figures/convergence.pdf.
Element order configurable + first-run confirmAn --order flag (default 1); on each --convergence run, prompt to confirm the order, persisting the choice in .element_order_confirmed.
Sanity check on every solveReport max(u_h) per level. The manufactured solution attains max = 1; values far from this signal trouble.

For P1 you should see L² rate → 2 and H¹ rate → 1 in the table, with the data lines parallel to the dotted reference lines on the plot.

Exercise 2 · framing

Why a plan first for a broken convergence test

The L-shape corner singularity is one of the most-taught problems in FEM, so the assistant has a strong textbook answer ready. Plan mode is what lets you check whether that textbook matches this code before any of it is written.

DecisionWhat plan mode forces you to settle in advance
Diagnostic chosenWhat "where the error is" actually means — element-wise |u_h − u_exact| vs. a residual indicator vs. eyeballing a plot. Skip this and you are refining without evidence.
Refinement mechanismWhether the implementation is one line or a hundred. MeshHierarchy + coordinate transformation, NetgenHierarchy with a marker, PETSc-level cell refinement — pick one before code is written.
Grading parametersWhat "graded toward the corner" actually means: a radius, a depth, an exponent, a marking criterion. Defaults pinned in the plan, not invented during implementation.
Out-of-scope disciplineKeeps the assistant from rewriting the rates harness while it is in there. Lose the harness and you lose the before/after comparison.
Visible before/afterThe same harness runs both times; that is the test. The plan has to leave the rates loop intact.

Exercise 2 · principle

Claude is a tool. You are the scientist.

On a classical pedagogical problem like this one, the assistant has seen the textbook answer many times and will draft it quickly. That cuts both ways: it tends to compress the diagnosis step — "we know it is the corner" — or to substitute a different refinement strategy than the one you asked for, because that is the variant it has seen more often.

Your job in plan mode is to decide how much structure to impose so the plan actually matches the problem in front of you. The "only the mesh is in scope" line in the prompt is one example of that structure. When you find yourself imposing the same structure session after session, promote it into your CLAUDE.md; that is what the file is for.

Claude Code becomes risky the moment you start trusting it to make major scientific decisions without checking. The pattern-match is fast and often plausible, and on a familiar problem it is very hard to tell a textbook reflex from real reasoning. The check has to come from you.

Exercise 2 15 min

Run the exercise

  1. cd exercises-pde/02-planning && claude; start the Firedrake container per ../01-claude-md/INSTALL.md.
  2. In the container, run python3 laplace_lshape.py. Note the L² rate stuck around 4/3 and the H¹ rate around 2/3 — far from the 2.0 and 1.0 the footer line says to expect.
  3. Press Shift+Tab twice to enter plan mode (look for the indicator at the bottom of the screen).
  4. Paste the prompt and submit. Do not approve yet. 
    The convergence in this code is suboptimal. Create a plan to
identify where the error is concentrated, then construct an a
priori mesh refinement strategy in the problem regions.

The polynomial order is fixed at P1; only the mesh is in scope.
  5. Read the plan against four highest-leverage questions: diagnostic mechanism? refinement primitive? grading parameters? what is out of scope? Push back on at least one.
  6. Approve. Watch the assistant edit laplace_lshape.py. Re-run; the L² rate should climb visibly toward 2.0.

Exercise 2 · debrief

What a good plan + implementation produces

A plan that survives review on this exercise has six sections. Each one closes off a class of drift that would otherwise surface during implementation.

Plan sectionWhat it contains for this exercise
GoalOne paragraph. Modify laplace_lshape.py so the study refines more densely near the corner; target L² rate toward 2.0 and H¹ toward 1.0; keep the rates harness, manufactured solution, BC, and P1 untouched.
Step-by-stepInspect the mesh → implement the element-wise error diagnostic → implement the grading transformation → wire it into the rates loop → run and verify.
Out of scopeA posteriori adaptive marking. Dörfler loops. hp-refinement. Alternative element families. Touching the manufactured solution, the BC, or the polynomial order.

A successful run shows the rates moving visibly in the right direction — not necessarily reaching the textbook constants. Closing the last fraction is FEM-tuning craft and out of scope for this exercise.

Exercise 2 · closing

What plan mode is really for

Plan mode produces text. The value is not the text — it is the questions the act of planning forces you to answer in advance. They are the same questions you would have hit during implementation, just in an order where you can still afford to redirect cheaply. On a problem the assistant knows well, those questions are also the ones the assistant most wants to skip; on a problem it has never seen, they are the ones you most need to spell out. Either way, the scientist is you.

Take the conventions you found yourself imposing back to your CLAUDE.md. The next exercise builds on that file again.
